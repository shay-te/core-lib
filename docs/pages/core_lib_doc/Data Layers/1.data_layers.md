As an app grows, database queries, business logic, and API glue get tangled together — one change requires editing five files. Core-Lib enforces a strict three-tier separation: **Data** (models and connections), **Data Access** (queries and storage operations), **Service** (business logic, caching, transformation). Each tier only talks to the tier below it, so changes stay local and code stays reusable.

The diagram below shows how a request flows from your host application down through the layers to the data source. Each box only talks to the one directly below it.

```
┌──────────────────────────────────────────────────────────────────────┐
│  HOST APPLICATION  (Flask / FastAPI / Django / ...)                  │
│    Routes, controllers, request handlers                             │
│                                                                      │
│    your_core_lib.user.get(user_id)                                   │
│    your_core_lib.user.create(...)                                    │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼  (synchronous Python call)
┌──────────────────────────────────────────────────────────────────────┐
│  YourCoreLib  (entry point, inherits core_lib.CoreLib)               │
│    self.user      →  UserService                                     │
│    self.<feature> →  <Feature>Service                                │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│  Service layer  (data_layers/service)                                │
│    - Business logic                                                  │
│    - Caching (@Cache)                                                │
│    - Transform return data to dict (@ResultToDict)                   │
│    - Composes one or more Data Access classes                        │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│  Data Access layer  (data_layers/data_access)                        │
│    - All queries / storage operations                                │
│    - Rule validation on create/update                                │
│    - Data-level exception handling                                   │
│    - e.g. CRUDDataAccess, CRUDSoftDeleteDataAccess                   │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│  Data layer  (data_layers/data)                                      │
│    - Entities (SQLAlchemy ORM + mixins like SoftDeleteMixin)         │
│    - Connection factories (SQLAlchemy / Mongo / Solr / Neo4j / ES)   │
│    - Alembic migrations                                              │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│  Data source  (Postgres / MySQL / SQLite / Mongo / Solr / Neo4j /...)│
└──────────────────────────────────────────────────────────────────────┘
```

### The Data Layers

1. `Data` Layer
2. `Data Access` Layer
3. `Service` Layer



## Data Layer

The `Data` layer defines low-level assets that are used to connect to various third-party sources. For example connection, entities, migration, mapping, etc.

#### Responsibilities: 

- Define models/entities/connection/migration used by data source

#### Example define a database entity under the DB connection

##### <project_dir>/data_layers/data/db/user.py

```python
from core_lib.data_layers.data.db.sqlalchemy.base import Base
from core_lib.data_layers.data.db.sqlalchemy.mixins.soft_delete_mixin import SoftDeleteMixin
from sqlalchemy import Column, Integer, VARCHAR


class User(SoftDeleteMixin, Base):
    __tablename__ = 'user'

    id = Column(Integer, primary_key=True, nullable=False)
    email = Column(VARCHAR(length=255), nullable=False)
    ...
```

Inheriting from both `SoftDeleteMixin` and `Base` gives the entity soft-delete columns (`created_at`, `updated_at`, `deleted_at`) automatically. `SoftDeleteMixin` must come before `Base` in the inheritance order so its methods take precedence.

## Data Access Layer

Encapsulate and expose access methods to the `Data` layer.

#### Responsibilities: 

- Define `APIs` to access the `Data` Layer.
- Using RuleValidator to validate update and create data.
- Exception handling happening in the data level.
- Optimization and speed of data fetching.

```python
class UserDataAccess(DataAccess):

    def __init__(self, db_session: SqlAlchemyConnectionFactory):
        self.db_session = db_session

    def get(self, id: int):
        with self.db_session.get() as session:
            return session.query(User).get(id)
```


## Service Layer

The `Service` layer provides an API that its users can access.  And it will use  `DataLayer`/Other `Services`/`Connection` to do so. 

#### Responsibilities: 

- Business Logic

- Transform return data to `dict`

- Caching

```python
from core_lib.cache.cache_decorator import Cache
from core_lib.data_transform.result_to_dict import ResultToDict
from core_lib.data_layers.service.service import Service

CACHE_KEY_USER = 'key_user_{user_id}'


class UserService(Service):

    def __init__(self, user_data_access: UserDataAccess, user_friends_data_acccess: UserFriendsDataAccess):
        self.user_data_access = user_data_access
        self.user_friends_data_acccess = user_friends_data_acccess

    @Cache(CACHE_KEY_USER)
    @ResultToDict() 
    def get(self, user_id: int):
        user = self.user_data_access.get(user_id)
        if user.something:
            user.friends = self.user_friends_data_acccess.get_user_friends(user_id)
        return user
```
